    Structure (mirrors `compile_bitcode` binding, lines 1041-1066):

    1. Lambda signature: `[](CUDACompiler &compiler, const std::vector<CudaFuncRequest> &requests) -> py::tuple`
    2. Call: `auto [bitcode, error, results] = compiler.inferReturnTypes(requests);`
       - Note: `bitcode` will be empty (no emit), `error` will be empty on success
    3. On success (`error.empty()`):
       - Build `py::list pyResults`
       - For each `r` in `results`: `py::list pyRetTypes` populated with `py::cast(tp)` for each `TensorParameter` in `r.ReturnTypes`
       - Per-result tuple: `py::make_tuple(r.Symbol, py::str(""), pyRetTypes, py::list())`
         The second field (MangledName) is empty string — no codegen (D-01).
         The fourth field (ExtractorMangledNames) is empty list — no extractors at inference time.
       - Append to `pyResults`
       - Return `py::make_tuple(true, py::bytes(bitcode), py::str(""), pyResults)`
    4. On error: `return py::make_tuple(false, py::none(), py::str(error), py::list());`
    5. Binding: `py::arg("requests")` — single argument, same as compile_bitcode.

    Key differences from compile_bitcode binding:
    - Calls `compiler.inferReturnTypes(requests)` instead of `compiler.compileBitcode(requests)`
    - MangledName field is always `py::str("")` (empty)
    - ExtractorMangledNames field is always `py::list()` (empty)
    - Returned bitcode bytes are always empty
    - Same 4-tuple return structure: `(ok: bool, bitcode: bytes, error: str, results: list)` — preserving the convention for consumer consistency

    The Python consumer accesses result data as: `r.return_types` (from `CudaFuncResult` line 1015), where each `TensorParameter` has `.type` (ScalarType enum), `.shape` (list of ints), `.layout_shape`, `.reg_basis`, `.lane_basis`, `.warp_basis`, `.n_warps`.

    No changes to existing bindings (constructor, parse, compile_bitcode).
