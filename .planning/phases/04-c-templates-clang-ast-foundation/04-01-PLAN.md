<objective>
Define the CUDA C++ device-side SharedLinearLayout and SharedTensor templates in tt_plugin.cu, declare the
SharedLayoutInfo/SharedTensorParameter C++ structs and TypeBuilder/TypeInspector extensions in clang_compiler.h,
and expose llvm.SharedTensorParameter via pybind11 in llvm.cc.

Purpose: Establish the type system and data-carrying structures that the remaining Phase 4 plans build upon — the
device templates are the source of truth for clang AST construction/inspection, the C++ structs carry layout data
across the Python/C++ boundary, and the pybind binding lets Python tests construct and inspect shared-tensor
parameters.

Output: Working device templates, C++ header declarations, and a Python-importable SharedTensorParameter class.
</objective>

  <action>
  Insert new device-side template definitions in tt_plugin.cu AFTER line 99 (end of existing Tensor definition, before the `// ==== END OF DEFINITIONS ====` marker).
  Per D-01: define flat top-level carriers. Per D-02: each basis row is a length-RANK IntTuple vector.
  Per D-03/D-04: SharedTensor holds zero-length T data[] with operator() returning T&. Per D-05: test function uses SharedTensor& params. Per D-06: evaluate() fully implemented — blockIndices accepted but unused (Phase 4 passes `{}`).

  A. OffsetBases<RANK> NTTP carrier struct (D-01, D-02):
     - Template parameter: uint32_t RANK
     - Field: `IntTuple<RANK> Dims[]` with flexible-array-like size; actual array length set via class specialization
     - constexpr default constructor (zeros Dims)
     - constexpr constructor from `std::initializer_list<IntTuple<RANK>>`
     - Mirror the existing `BasisGroup` pattern at tt_plugin.cu:38-44 but as a standalone top-level template (NOT nested inside TensorLayout)

  B. BlockBases<RANK> NTTP carrier struct:
     - Identical structure to OffsetBases<RANK>
     - Separate type for type-safety (so SharedLinearLayout template args are distinguishable)

  C. SharedLinearLayout<OffsetBases OB, BlockBases BB, uint32_t Alignment> template (D-01, D-06):
     - NTTP parameters: `OB` of some OffsetBases specialization, `BB` of some BlockBases specialization, `Alignment` as uint32_t
     - static constexpr uint32_t RANK = decltype(OB)::RANK (derived from the OffsetBases type)
     - static constexpr uint32_t Align = Alignment
     - static constexpr IntTuple<RANK> evaluate(const IntTuple<RANK>& logicalIndices, const IntTuple<RANK>& blockIndices):
       - offset contribution: for each row i in OB.Dims, if the corresponding encoding bit is set, XOR-add OB.Dims[i] via IntTuple::operator+
       - block contribution: same pattern for BB.Dims
       - Combined result = offset_contrib + block_contrib (element-wise XOR via IntTuple::operator+)
       - The encoding maps: each basis row corresponds to a bit position in an internal linear index derived from logicalIndices.
         Phase 4 encoding: the i-th row of OB.Dims contributes when bit i of a "serialized" logical index is set.
         Specifically: flatten logicalIndices to a linear index (e.g., row-major over shape dims), then apply BasisGroup-style bit-check per row.
       - blockIndices accepted but unused this phase (D-06: "Phase 4: blockIndices = {}")
       - IMPORTANT: this formula must be bit-identical to MLIR LinearLayout({offsetBases,blockBases},outDims).apply() on a 1D output byte-offset. Verify by studying the Triton LinearLayout::apply implementation.
     - No `REG_SIZE` member (SharedLinearLayout is NOT a register layout — it computes byte offsets into shared memory)

  D. SharedTensor<T, TShape, TLayout> template (D-03, D-04, D-05):
     - Template params: typename T (scalar type), typename TShape (Shape<DIMS...>), typename TLayout (SharedLinearLayout<...>)
     - `T data[]` — zero-length array at struct end (D-03: aliases external shared memory, no owned storage; lowers cleanly to ptr addrspace(3) in Phase 6)
     - `__device__ T& operator()(uint32_t logical_idx)` — single-index variant for 1D tensors:
       * Construct IntTuple<RANK> from logical_idx (spread across dimensions per TShape)
       * Call TLayout::evaluate(indices, {}) to get IntTuple<RANK> byte-offset components
       * Compute 1D byte index from IntTuple components (dot with stride)
       * Return data[byte_index] as T& (D-04: mutable reference = read+write)
     - The return type `T&` is critical — the executor must verify the function signature literally returns `T&` (not `T` or `const T&`)

  E. Test-only device function (D-05 exercise):
     - After the template definitions, add:
       ```
       template<typename T, uint32_t N, typename TLayout>
       __device__ void write_shared(SharedTensor<T, Shape<N>, TLayout>& shm, T val) {
           shm(0) = val;
       }
       ```
     - Uses `SharedTensor<...>&` parameter (D-05)
     - Exercise both operator() read (none yet) and operator() write (assignment to T&)
     - This function is consumed by Plan 04-03's pytest harness (GPU-free, compiled via SuspendedCudaCompiler but never launched)
  </action>
