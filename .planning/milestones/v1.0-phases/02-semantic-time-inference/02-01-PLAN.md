    2. Inside the `Tensor<T, TShape, TLayout>` struct (lines 81-84), after `T data[TLayout::REG_SIZE];` (line 83), add an implicit conversion constructor:
       ```
       template<typename T2, typename TShape2>
       Tensor(const Tensor<T2, TShape2, PlaceholderLayout>& other) {
           static_assert(std::is_same_v<T, T2>, "dtype mismatch in PlaceholderLayout conversion");
           static_assert(std::is_same_v<TShape, TShape2>, "shape mismatch in PlaceholderLayout conversion");
           #pragma unroll TLayout::REG_SIZE
           for(uint32_t i = 0; i < TLayout::REG_SIZE; i++)
               data[i] = other.data[i];
       }
       ```
    3. No other changes to tt_plugin.cu. The existing device functions (elementwise_add, intra_warp_add_sibling, add_bias, reduce, split_add) and type framework (Shape, TensorLayout, Layout, BasisGroup, IntTuple) remain untouched.

    Why this works (D-06): When the clang compiler builds `Tensor<scalar, Shape<dims...>, PlaceholderLayout>` as the argument type and calls `LookupFunction`, the template deduction engine discovers the implicit conversion and matches against any `Tensor<scalar, Shape<dims...>, ConcreteLayout>` signature. Since dtype+shape are layout-independent in this framework, deduction succeeds and yields the correct scalar type and shape per D-06. Multiple candidates disagreeing on dtype/shape would result in lookup failure (ambiguous overload) — the existing `LookupFunction` error path handles this.

    At the top of `CUDACompiler::BuildTensor` (line 705), after the `clang::QualType Result;` inside the TaskQueue lambda, add a placeholder-mode early-return path. When all three basis vectors are empty AND `N_WARPS == 0`, skip the concrete Layout building and directly construct `Tensor<T, Shape, PlaceholderLayout>` instead:

    <!-- planner-discipline-allow: N_WARPS == 0 -->
    - Detect: `if (Param.Layout.N_WARPS == 0 && Param.Layout.RegBasis.empty() && Param.Layout.LaneBasis.empty() && Param.Layout.WarpBasis.empty())`
    - In placeholder mode: build the Shape type via `helper.Builder.buildShape(Param.Shape)` (same as line 709), get the scalar QualType via `getQualTypeFromScalarType(helper.Builder.Ctx, Param.Type)`, then look up `PlaceholderLayout` from the AST: `auto *Lookup = Ctx.getTranslationUnitDecl()->lookup(&Ctx.Idents.get("PlaceholderLayout"));`. If the lookup succeeds, `auto *TD = llvm::dyn_cast<clang::TagDecl>(*Lookup.begin());` and `auto PlaceholderQualType = Ctx.getTagDeclType(TD);`. Build the result via `Result = helper.Builder.BuildTensor(scalarQualType, Shape.type, PlaceholderQualType);`. Return from the lambda (or set Result and skip the rest).
    - If lookup fails (PlaceholderLayout not in the parsed source), fall through to the existing concrete-layout code path — this preserves backward compatibility for .cu files that don't define PlaceholderLayout.
    - The existing concrete-layout code (lines 710-733) remains unchanged; the placeholder branch is inserted before it.

    **Change B — clang_compiler.h: inferReturnTypes declaration** (D-01)

    After the `compileBitcode` declaration (lines 348-349), add:
    ```cpp
    // D-01: Inference-only — runs type inference (Phase 1 of compileBitcode)
    // on an already-parsed compiler without emitting LLVM bitcode.
    // Returns per-request CudaFuncResult with ReturnTypes populated;
    // MangledName/ExtractorMangledNames are empty (no codegen).
    std::tuple<std::string, std::string, std::vector<CudaFuncResult>>

    ```
    Placed inside the `CUDACompiler` struct, before the closing `};` (line 350).

    **Change C — clang_compiler.cc: inferReturnTypes method** (D-01)

    Add the new method body after `EmitFinalModule` (after line 804) and before the `compileBitcode` method. The method copies Phase 1 of `compileBitcode` (lines 830-890) exactly — per-request `BuildTensor` → `LookupFunction` → `EvaluateFunctionReturnType` — but stops before Phase 2 (codegen: `InstantiationFunction`) and Phase 3 (emit: `EmitFinalModule`).

    Method signature matches `compileBitcode`: `std::tuple<std::string, std::string, std::vector<CudaFuncResult>> CUDACompiler::inferReturnTypes(const std::vector<CudaFuncRequest> &requests)`.

    Body structure:
    1. `std::vector<CudaFuncResult> results;`
    2. For each request (identical to lines 831-890):
       - Build `argTypes` from `req.ParamTypes` — `BuildTensor` for TensorParameter, `getQualTypeFromScalarType` for ScalarType (with TaskQueue/SwitchTo coroutine pattern)
       - `auto *FD = this->LookupFunction(req.Symbol, argTypes);` — fail-fast with error return on nullptr
       - `auto ret = this->EvaluateFunctionReturnType(FD);` — extract `TensorParameter`s from single-return or `TupleType`
    3. Build `CudaFuncResult r; r.Symbol = req.Symbol; r.ReturnTypes = std::move(retParams);` — no `MangledName`, no `ExtractorMangledNames`, no extractor lookup (no codegen, no tuple-element extractors needed at inference time)
    4. `results.push_back(std::move(r));`
    5. Return `{"", "", results};` — empty bitcode, no error, populated results.

    **Coroutine discipline:** Every `BuildTensor`, `LookupFunction`, `EvaluateFunctionReturnType` call already embeds the `TaskQueue.emplace` + `InvocationContext->SwitchTo` pattern — no new coroutine plumbing needed. The `EvaluateFunctionReturnType` call uses `this->` (lines 779-791) — use `this->EvaluateFunctionReturnType(FD)` directly.
